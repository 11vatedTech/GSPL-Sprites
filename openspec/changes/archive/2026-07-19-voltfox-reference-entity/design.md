## Context

GSPL Sprites is a deterministic sprite-authoring library supporting 2D, 2.5D, and 3D projections via a shared semantic model (seed → IR → package). All subsystems compile and tests pass under both MSVC and MinGW GCC. The `examples/voltfox.sprite` seed file exists but no visual-set or end-to-end build artifact validates the full pipeline. The synthesis API (`synthesize_unified_entity`) can generate all projections from a palette and rig, but no integration test wires the complete flow: seed → parse → compile → render → synthesize → build → verify.

## Goals / Non-Goals

**Goals:**
- A complete, buildable Voltfox reference entity that exercises the full seed → package pipeline
- An integration test (CMake `add_test`) that builds and self-verifies the Voltfox package
- Authored visual-set (`.txt` + PNGs) for Voltfox, loadable by `load_authored_visual_set`
- CI-reproducible end-to-end correctness check

**Non-Goals:**
- High-quality art assets (PNGs are procedural/generated for testing, not hand-drawn)
- Performance benchmarks
- The full living-runtime vertical (Phase D scope: living-runtime + replication + event routing)

## Decisions

1. **Visual-set PNGs generated by code, not committed bitmaps** — PNGs under `examples/` would bloat the repo and rot. Instead, a test helper or synthesis step generates them via `make_image()` + `encode_png()` at test time. This also validates the PNG encode/decode round-trip.

2. **Integration test lives in `tests/voltfox_integration_tests.cpp`** — Separate from the existing `tests/` suite to keep the reference-entity integration test as a distinct target. Follows the existing naming convention (`gspl_sprites_voltfox_integration_tests`).

3. **Build uses seed-only path (`build_package(seed, output)`)** — The seed-only path triggers the synthesis pipeline internally. Adding a visual-set would test the `build-visual` path too, but that requires authored frame assets. We do both: one test for seed-only synthesis, one for seed+visual-set.

4. **Visual-set content mirrors the voltfox seed** — The visual set defines 2D frames matching the animation clips, directions, and layers described in `voltfox.sprite`. Channel maps for depth and effects validate the channel-map pipeline.

## Risks / Trade-offs

- [Generated PNGs are deterministic] → Use fixed random seed, same palette, same geometry. The `encode_png`/`decode_png` round-trip must be lossless; tested separately.
- [Integration test duplicates unit-test coverage] → Acceptable. The purpose is pipeline integration, not unit coverage. One test per representation (2D, 2.5D, 3D) plus the combined pipeline.
- [visual-set format evolution] → The visual-set manifest schema (`gspl.visual-set/0.1`) is fixed for this change. Schema migration is future work.
